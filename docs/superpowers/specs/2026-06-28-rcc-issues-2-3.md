## Summary

The v1 design (`2026-04-23-rcc-design.md`) optimized `rcc` for *interactive*
typing. Issues #2 and #3, both filed from downstream automation (`serving-stack`
/ `snapper`, and a non-SLURM CloudLab sweep), collect the friction that appears
when `rcc` is *scripted* instead. This doc covers everything added to close
those two issues. Every item below is implemented, tested (228 unit tests), and
documented in the README.

The recurring root cause across both issues: glue that consumers had to
hand-roll — deletion-safety comments in `rccignore`, `bash -lc` prefixes,
`shlex.quote` env injection, `squeue` poll loops, tmux orchestration, manual
`ssh -L`, `sed` over `rcc config`. The work below moves that glue into the tool.

## Design decisions

### Sync safety (issue #2 P1)

**Split `--delete` into two flags.** `--delete` is now *bounded* sync: rsync
`--delete`, which only removes files inside the non-ignored transfer scope
(excluded files are spared). The dangerous `--delete-excluded` semantics moved
to a distinctly-named `--mirror` flag. This matches the issue's "make the
default non-destructive, require explicit opt-in for the dangerous semantics"
and rsync's own prior art.

**Protect-list lives in the tool, not the user's memory.** `--keep-remote GLOB`
(repeatable, CLI) and profile-level `keep_remote = ["logs/", "cache/", ...]`
compile to rsync `--filter=protect <glob>` rules, which survive **both**
`--delete` and `--mirror`. This is the "rccignore-with-teeth" the issue asks for
— job-owned paths (`logs/`, `*.safetensors`, `last_service.env`) are protected
even when the user runs a mirror.

**Deletions can't hide in dry-run.** `--dry-run` parses rsync's
`--itemize-changes` output and bucketizes it into a `Would DELETE:` section
(distinct from `Would SEND:`/`Would RECEIVE:`), so a destructive transfer is
visible at a glance rather than buried in `*deleting   ...` / `f.f.....` lines.

### Scoped transfers (issue #3 #1)

`rcc push <subpath>` / `rcc pull <path> [dest]` scope the rsync to a subtree.
**Scoped transfers bypass `.rcc/rccignore` by default** (the issue's explicit
ask: "let it override rccignore" — the use case is pulling a results dir that
lives in rccignore so it isn't pushed back). Whole-project transfers still
honor rccignore unless `--no-ignore`. Path-escape validation rejects `..`
segments and absolute escapes. Scoped transfers require rsync (not the
paramiko fallback) to keep the semantics tractable; a missing rsync raises
`MissingDependencyError` with a clear message.

### `run` tee (issue #2 P1, "streaming vs exit-code")

`subprocess.run` can't both stream and capture. `run_remote_tee` uses `Popen` +
two reader threads: each thread writes live bytes to the real `sys.stdout`/
`sys.stderr` file objects *while* accumulating into a buffer, then returns a
`CompletedProcess`. Exposed as `rcc run --result-json PATH` — output streams to
the TTY, and on exit rcc writes `{returncode, stdout, stderr, command}` to
`PATH`. A wrapper using `subprocess.run(argv)` (streaming) gets live output,
then reads `PATH` for the authoritative text + exit code. (`--result-json` is
file-path-only, not stdout, so it never conflicts with the streamed output.)

### Detached runs / `rcc bg` (issue #3 #2)

A new `rcc bg` subgroup mirrors the `rcc job` UX for non-SLURM hosts, backed by
tmux so runs survive disconnects. `rcc run --detach [--name N]` launches a tmux
session named `rcc-<N>`; the session's wrapper script is:

```bash
set -o pipefail
mkdir -p -- "$REMOTE_DIR/.rcc-runs"
<env exports>
cd -- "<resolved cwd>"
( set -euo pipefail; <command> ) 2>&1 | tee -- "$REMOTE_DIR/.rcc-runs/<name>.log"
printf '%s' "${PIPESTATUS[0]}" > "$REMOTE_DIR/.rcc-runs/<name>.status"
```

So `tee` captures combined stdout+stderr to a log file (and passes it to the
tmux pane, so `attach` sees it live), and `PIPESTATUS[0]` records the command's
real exit code — verified against real bash for success, failure (`exit 42`),
and shell-pipeline paths. `bg wait` polls `tmux has-session`, then reads the
status file → exits with the run's exit code (124 on timeout).

State under `remote_dir/.rcc-runs/<name>.{log,status}` survives disconnects and
pulls back with `rcc pull`. `tmux` is auto-detected (exit 127 + hint if
missing). `bg logs`/`attach`/`wait`/`stop` default to the sole running session
when the name is omitted. Names are sanitized for tmux (`.`/`:` → `-`).

### Slurm lifecycle (issue #2 P2)

`rcc job` wraps sbatch/squeue/sacct/scancel so the user never shell-quotes a
`--format=` value, a `$USER`, or a pipeline. `job wait` closes the submit→monitor
loop: it polls `squeue -j ID -h -o %T` until the job leaves the queue, then
classifies the final state via `sacct -n -P --format=State,ExitCode` — exits `0`
on COMPLETED, the job's recorded exit code on failure, `124` on timeout. `--on
STATE` returns early (e.g. once RUNNING). Unlike `sbatch --wait`, it surfaces the
outcome without minutes-long silence.

### Structured output / `--json` (issue #2 P2)

The parser is the fragile bit, so it is designed defensively:

- **`job list --json`** uses squeue with `--format` of explicit `%`-codes joined
  by `|` under `-h` (no header). rcc defines *both* the `%`-code order and the
  paired field-name list, so parsing never depends on `--parsable` availability.
- **`job status --json`** uses sacct's well-established `--parsable` and
  **keeps the header**, reading field names dynamically — survives sacct
  renaming/reordering fields across versions.
- Both feed one pure, exhaustively-tested `parse_rows()` function. Each row with
  a `State` gains an `ok` flag; each row with an `ExitCode` of the form
  `RETURN:SIGNAL` gains parsed integer `exit_code`/`signal` fields (so an
  OOM-killed step surfaces as `exit_code: 137, signal: 9`).
- `config --json` and `status --json` likewise return stable JSON objects.

### Richer profiles (issue #2 P3)

`Profile` gained: `env` (dict; scalar numerics coerced to strings so
`PORT = 8080` works), `proxy_jump`, `identity_file`, `keep_remote`, `tunnel`
(`{remote_port, local_port?, remote_host?}`). These are parsed from TOML and
threaded into the SSH arg builder (`-J`/`-i`), the `run` env layering, the
`--mirror` protect-list, and `tunnel` defaults respectively.

## How to use it

> The README is the canonical, copy-pasteable reference; this section is a
> condensed tour. All examples assume `rcc init` has been run and the profile's
> `host`/`remote_dir` are set.

### Safe vs dangerous sync, with a protect-list

```bash
rcc push --delete                          # bounded sync: spares excluded files
rcc push --mirror                          # DANGEROUS: also deletes excluded files
rcc push --mirror --keep-remote 'logs/'    # mirror, but never touch logs/
rcc push --delete --dry-run                # deletions shown in a "Would DELETE:" section
```

Profile-level (in `.rcc/config.toml`):

```toml
[profiles.gpu]
keep_remote = ["logs/", "cache/", "*.safetensors", "last_service.env"]
```

### Selective transfer (pull just the results)

```bash
rcc push jobs/sweep                        # push one subtree (bypasses rccignore)
rcc pull jobs/sweep/                       # pull one subtree (bypasses rccignore)
rcc pull jobs/sweep/ ./out/                # ...reconstructed under ./out/
rcc push --no-ignore                       # whole project, ignore rccignore
rcc push --include '*.bin'                 # extra include globs
```

### Run with env, cwd, tee, or detached

```bash
rcc run --env EPOCHS=10 --env GPU=0 -- python train.py
rcc run --env-file .env -- python serve.py
rcc run --cwd /scratch/run -- bash job.sh
rcc run -s 'a && b | c'                    # shell snippet (pipelines, $vars, quotes survive)
rcc run --result-json /tmp/r.json -- srun python train.py   # tee: live + captured
```

### Detached runs (non-SLURM)

```bash
rcc run --detach --name sweep -- python train.py
rcc bg ps                                  # list rcc sessions
rcc bg logs sweep -f                       # tail the captured log
rcc bg attach sweep                        # attach to the live tmux session
rcc bg wait sweep                          # block; exit code = run's exit code
rcc bg stop sweep                          # kill it
```

### Port-forwarding

```bash
rcc tunnel                                 # uses profile [tunnel] defaults
rcc tunnel --remote-port 8080 --remote-host head01 --local-port 18080
```

Profile default:

```toml
[profiles.gpu.tunnel]
remote_port = 8080
local_port = 18080        # optional; defaults to remote_port
remote_host = "localhost" # optional; set to a compute node to reach a service there
```

### Slurm lifecycle

```bash
rcc job submit train.sh --extra-env EPOCHS=10
rcc job submit train.sh -W --dependency afterok:524614   # block + chain
rcc job list                               # squeue for your user
rcc job list --json                        # structured records
rcc job status 524614                      # sacct -j <id>
rcc job status 524614 --json               # structured (main + steps, with exit_code/signal/ok)
rcc job tail 524614 -f                     # tail -f slurm-524614.out
rcc job wait 524614                        # poll until done; exit = job's code
rcc job wait 524614 --on RUNNING           # ...or until it reaches a state
rcc job cancel 524614
```

### Machine-readable config / status

```bash
rcc config --json                          # resolved profile as JSON
rcc config --get host                      # single value, unquoted
rcc config --get env.TRITON_CACHE_DIR
rcc config --get tunnel.remote_port
rcc status --json                          # {controlmaster_open, host}
```

## Testing strategy

Business logic stays in pure modules (`sync.py`, `ssh.py`, `slurm.py`, `bg.py`,
`config.py`); `commands/*.py` are thin typer wrappers. Tests exist at both
levels and mock the SSH/rsync boundary (no network). Notable additions:

- **Real-rsync integration tests** (`tests/test_sync_rsync.py`) exercise scoped
  transfer semantics and `keep_remote` protection under `--mirror` against a
  real `rsync` binary — cheaper and more faithful than mocking rsync's grammar.
- **The detached-run wrapper script is validated against real `bash`** for
  success, failure (`exit 42`), and shell-pipeline paths, proving the
  `PIPESTATUS`/`tee`/status-write semantics independently of tmux.
- **The Slurm JSON parser** is tested exhaustively against realistic
  squeue/sacct fixtures (header+rows for sacct, known-column rows for squeue),
  plus the enrichment (`exit_code`/`signal`/`ok`) and ragged-row padding.

228 tests pass; `ruff check` and `ruff format --check` are clean.
